哈希算法

  • 哈希算法生成的数据格式被称为散列，散列不是一个可逆的过程。而之前说过，Base64只是编码，顾名思义，编码只要知道规则，是可以解码的，没有安全性可言。
  • 所以讲密码哈希一下，是没法看出来原始密码是什么的。
  • 这边可以扩展，进一步研究的是，如果进行攻击的话，称为字典攻击，就是猜密码，将常用的都哈希一下，去对比。
  • 哈希算法有很多种，比如MD5是一种，虽然之前说就是提取摘要的。SHA-512是另一种。

加密

  • Encryption lies in between hashing and encoding 这句没看明白
  • 加密算法可以分为对称加密和非对称加密两种

对称加密

  • 主要来自参考链接一， 对称加密的话，文章中用一个例子来讲解：“移位”。比如有数字"123"，对其进行加密，都增大3，成为“456”，就是对称加密了。
  • 其中的数字3就是Encryption key (symmetric key)。对称加密的时候，知道key的话，就可以进行解密了。
  • Anyone who knows the encryption key and can apply the reverse of Caesar’s algorithm and retrieve the original Plaintext.
  • 有此可知，在TLS中使用简单的对称加密不是很安全的。毕竟知道Encryption key就可以进行解密了。对称加密一代代发展，现在也有比较复杂的算法，比如Triple Data Encryption Algorithm，后面可以再进一步研究。
  • 从过程可以知道，key是非常关键的。但是由于是对称加密解密，双方必须都知道约定的值，那么第一步就是传递key，而这个时候是没法加密的。所以就出现了不对称加密。So we need a secure mechanism to exchange the keys without an attacker actually seeing it. There comes the role of Asymmetric Key Encryption.

不对称加密

  • 不对称加密始于RSA算法。简单的解释就是，用不同的key来进行加密和解密。
  • 公钥用来给对方使用，比如A想给B发消息，那么A需要使用B的公钥来加密自己的消息。
  • 私钥用来解密收到的消息。比如A收到了一条消息，那么A应当使用自己的私钥来解密收到的消息
  • 所以RSA算法有几个比较重要的组成，私钥，公钥，加密算法和解密算法
  • 那么一个严肃的问题就是，不同的钥匙进行计算的消息，怎么能够解开呢？接下来就是数学课的时间了。
    • 参考链接：
      • 带你彻底理解RSA算法原理
      • RSA 算法的加密原理是什么,知乎问答
    • 互质数： 只有一个公因子为1的数。比如6和9，都有公因子为3，那么就不是互质数；而7和9只有公因子为1，就是互质数。
    • 欧拉函数：φ(n)。 小于或等于n的正整数中与n互质的数的数目。
    • 三个横线的奇怪等号是恒等于。
    • 生成公钥：随机选取两个质数p1、p2，n=p1*p2，再随机选取一个整数e，e与φ(n)互质。公钥就是(e, n)。
    • 根据欧拉函数定理，φ(n) = (p1-1)*(p2-1)
    • 费马定理：若p是素数，a与p互素，则a^(p-1）≡1 （mod p） 没看懂这个费马定理
    • 加密算法：假设m为需要加密的信息，那么(m^e) mod n就加密好了，生成加密后的信息c
    • 解密算法： 解密算法其实和加密算法的公式是一模一样的，只是这里的m和e换成c和d，成为(c^d) mod n。相当于采用公式都是(x^y) mod n，所以其实使用的公式是一样的
    • 那么这里的(d, n)就被成为私钥。那么如何得到d呢？烧脑的数学就来了。果然数学系才有前途啊 😂 实在是颇为烧脑啊。
    • 各个参考链接中均有实际的例子，我这里就不再烧脑来看了
    • 参考链接：
      • 如何获得d
(c^d) mod n = ((m^e)mod n)^d mod n = (m^ed) mod n = m
(m^φ(n)) mod n≡1 又 1^k≡1, 所以(m^k*φ(n)) mod n≡1 ，所以(m^(k*φ(n)+1)) mod n≡m
结合以上两条，ed=k*φ(n)+1 , d=(k*φ(n)+1)/e

TLS版本

  • SSL3.0 和 TLS1.0 是一个东西。嗯，微软干的。微软虽然是一个伟大的公司，但是这个干得不好，相当不好，至少我本人觉得不好。实在是除了增加大家的混淆以外，没有任何意义。嘛，我们自己的产品也经常干这种事情。通常呢，是因为自己认为发生了跨时代的改变，希望让别人也明白。
  • 目前最流行的版本是TLS1.2，最新的版本是TLS1.3。
  • java8中默认的是TLS1.2版本。java11里面支持的是TLS1.3
  • 各种web应用服务器的配置文件里面都可以配置支持或者不支持的版本。比如tomcat提供了参数:sslEnabledProtocols

TLS和OAuth

  • does-an-oauth-2-client-really-need-tls： 我自己有点搞不清楚这些的关系，这边有一个解释很是清晰，看了就明白了
    • 3.1.2.1. Endpoint Request Confidentiality If a redirection request will result in the transmission of an authorization code or access token over an open network (between the resource owner’s user-agent and the client), the client SHOULD require the use of a transport-layer security mechanism. 这个是Oauth官方文档里面的一段话。
    • 上面一段的意思应该是这样的：一个第三方认证的请求在传输authorization code或者access token的时候应当在传输层使用安全机制。这里的安全机制就是指TLS协议。

KeyStore和TrustStore

  • keystore和truststore从其文件格式来看其实是一个东西，只是为了方便管理将其分开
  • keystore中一般保存的是我们的私钥，用来加解密或者为别人做签名
  • truststore中保存的是一些可信任的证书，主要是java在代码中访问某个https的时候对被访问者进行认证的，以确保其实可信任的。一般为公钥
  • 只有HTTPS才需要这些吗？
